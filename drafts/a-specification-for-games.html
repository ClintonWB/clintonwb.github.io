A Specification for Games

Clinton bradford | Sun 12 September 2021

Game Specification

My previous attempts at designing games or platforms have been largely hard-coded. For the Firebase/React platform on which I built my Letter Jam, the game was provided largely free authority for modifications on a shared dictionary, but beyond that it was up to each individual game.

For the culmination project, however, I'd like to handle as much of the common code as possible in abstractions. In addition, I'd like to embed graphical assets into the file.

Over the years, I have seen a variety of formats for this sort of interactive content. LonCapa uses an (invalid) XML-based system, as does its spiritual successor EdX, to embed content. Geogebra uses an XML file, zipped together with any other visual assets, to represent its custom applets. Zip files can be transmitted over the internet and unpacked in browsers - a principle on which much of the modern web has been built - and would provide a handy container for the various components.

I have soured on XML over the years for anything other than turning into HTML - and another inspiration, Tabletop Simulator, uses a JSON-based format. I have used JSON extensively, and despite the parsing and security concerns, find it quite readable and versatile. A multi-JSON format, like JSON Lines, may also work well. So, taking Letter Jam as an example, a directory hierarchy might look like:

letterjam.zip:
    index.json
    objects:
        cards.json
        board.json
        proposing-round.json
        word-guessing-round.json
    images:
        card-back.svg
        card-front.svg
        player-flower-2.svg
        player-flower-3.svg
        player-flower-4.svg
        player-flower-5.svg
        player-flower-6.svg

So what goes in these files? Well, fundamentally, a board game is about components, the places those components can be, and actions you can take. In the case of Letter Jam, the components are - primarily - the cards. They can be in player's rows, in NPC holders, in the deck, or in the discard. And with them, you can spell words, or discard cards.

Each action has with it some conditions, which we would also have to represent. In previous versions, this would be in code. Here, I am tempted to set conditions in the JSON specification; for example, we could assign to each action an actor, that is, the player using it. We could check that actor's properties to see if they have a player clue remaining, unique to them, and check with the board's clues to see if there are any free.

{
    name = "Propose Clue",
    id = "propose-clue",
    conditions = [
        "game.phase === 'clues'",
        "game.free_clues>0 || actor.player_clues>0"
    ],
    effects = [
        "game.rounds += 1",
        "game."
    ]
}
